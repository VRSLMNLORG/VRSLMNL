## Общие правила работы в ветке dev

1) Перед тем как открывать Unity и делать изменения:

1. Открой GitHub Desktop.  
2. Убедись, что выбрана ветка dev (справа от названия репозитория).  
3. Нажми Fetch origin, чтобы проверить обновления.  
4. Если есть новые изменения — нажми Pull origin.

Это гарантирует, что ты работаешь с актуальной версией проекта, а не со старой.
Важно про Pull и твои изменения

> Перед “Pull origin” всегда делай коммит, если у тебя есть несохранённые изменения.  

Почему это необходи:  
Pull загружает обновления с сервера и объединяет их с твоей версией.  
Если ты изменял те же файлы, что и другие — Git может вызвать конфликт, и часть изменений может потеряться.  

Безопасный порядок действий:
1. Сделай Commit to dev своих изменений.  
2. Потом нажми Pull origin.  
3. Если GitHub Desktop покажет конфликт — реши его в Unity или код-редакторе.  

2) После завершения работы

Когда ты внёс изменения в Unity (новые сцены, скрипты, правки и т.д.):
1. Вернись в GitHub Desktop — появится список изменённых файлов.  
2. Убедись, что в списке только нужные файлы (твои изменения).  
   * Если Unity добавила .meta — оставь их, они нужны.  
3. В поле Summary (вверху слева) напиши короткое описание изменений.  
4. Нажми Commit to dev.  
5. Затем нажми Fetch origin, потом Pull origin (на случай, если кто-то уже запушил изменения).  
6. Наконец, нажми Push origin, чтобы отправить свои изменения.

Всегда делай “Pull” перед “Push”, чтобы не перезаписать чужие изменения.

3) Ответственность за свою часть проекта

Чтобы избежать конфликтов:
* Не редактируй сцены, ассеты и скрипты, над которыми работают другие участники.  
* Если нужно внести изменения — обсуди это в чате или в комментариях к коммиту.  
* Делай небольшие и осмысленные коммиты (например: «Добавлен прыжок игрока»).  
* Не коммить всё подряд («исправил всё сразу») — такие изменения трудно проверять.  


4) Проверка перед коммитом и пушем

Перед коммитом и перед пушем убедись, что:
1. Проект запускается в Unity.  
2. В Console нет ошибок (Errors).  
3. Ты сделал Fetch origin → Pull origin перед Push.  
4. При конфликтах — обращайтесь ко мне, решим вопрос.

Цель команды
Сохранять проект в рабочем состоянии, избегать конфликтов и облегчать совместную разработку.  
Чем чаще ты синхронизируешься с репозиторием, тем меньше шанс потерять чьи-то изменения.
